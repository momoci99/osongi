# 데이터 분석 필터 기능 리뷰 (지역/조합/기간)

- 작성일: 2026-04-19
- 범위: 데이터 분석 페이지 필터 (지역, 조합, 기간)
- 기준: 동작 정확성, 데이터 정합성, 유지보수성, 테스트 커버리지

## 요약

현재 필터 기능은 기본 동작은 안정적이지만, **조합 상수와 실제 데이터 불일치**로 인해 사용자가 의도한 결과를 얻지 못할 수 있는 문제가 있습니다. 또한 기간 필터 UX와 날짜 파싱 안정성 측면에서 개선 여지가 있습니다.

## 주요 이슈

### 1) [높음] 조합 상수와 실제 데이터 불일치

- 현황
  - `src/const/Common.ts`의 `REGION_UNION_MAP` 경북 목록에 `"예청"`이 포함되어 있음.
  - 실제 데이터에는 `"예천"`이 존재함.
  - 실제 데이터에는 `"영천"`도 존재하지만 상수 목록에는 없음.
  - **강원 목록에 `"의성"`이 잘못 포함되어 있음.** 실 데이터 전수 확인 결과 의성은 경북 전용 조합이며, 강원 데이터에 의성은 존재하지 않음.
    - 강원 실 데이터: `[강릉, 고성, 삼척, 양구, 양양, 인제, 홍천]`
    - `REGION_UNION_MAP` 강원: `[..., "의성"]` ← 오류
- 영향
  - 지역 선택 후 조합 목록에서 실제 조합이 누락되거나 오표기 노출.
  - 사용자 입장에서 필터 결과가 0건으로 보일 수 있음.
  - 강원 선택 시 존재하지 않는 "의성" 조합이 노출되어 선택 시 항상 0건.
- 근거
  - 상수: `src/const/Common.ts`
  - 데이터 예시: `public/auction-data/2024/10/20.json` (`"예천"`), `public/auction-data/2013/9/26.json` (`"영천"`, `"의성"` 경북 소속 확인)

### 2) [중간] 조합 옵션 생성이 하드코딩 맵에 의존

- 현황

- 영향

### 3) [중간] 기간 역전(startDate > endDate) 방지 UX 없음

  - DatePicker에 상호 제약(minDate/maxDate) 없음.
- 영향
  - 사용자가 역전 범위를 만들면 빈 결과가 나오고 원인 파악이 어려움.
- 근거

### 4) [낮음~중간] 날짜 파싱 브라우저 의존성 리스크

- 영향

## 사용성(UX) 관점 심층 분석

본 섹션은 기능 정합성 점검과 별개로, 사용자가 실제로 필터를 "빠르게 이해하고", "실수 없이 조작하며", "결과를 신뢰"할 수 있는지에 집중한다.

### 분석 관점

- 과업 성공성: 원하는 데이터 집합에 도달하는 데 필요한 단계 수와 실패율
- 인지 부하: 현재 상태를 이해하기 위해 사용자가 기억해야 하는 정보량
- 조작 효율성: 반복 사용 시 클릭 수, 스크롤 수, 재입력 빈도
- 오류 예방성: 잘못된 입력을 사전에 막는 정도
- 피드백 명확성: 시스템 반응이 사용자의 의도와 일치하는지
- 신뢰 형성: 결과가 왜 그렇게 나왔는지 설명 가능한지

### 예상 사용자 시나리오

- 시나리오 A: "강원 지역, 양양/강릉 조합, 최근 2주" 가격 추이 확인
- 시나리오 B: "경북 전체 조합, 특정 주간" 거래량 급증 여부 점검
- 시나리오 C: "올해 10월 1주 vs 전년 동기" 비교 분석
- 시나리오 D: 모바일에서 빠르게 기간 변경 후 결과 확인

현재 UI는 기본 흐름을 제공하지만, 아래 지점에서 과업 효율이 떨어질 가능성이 높다.

## UX 진단 결과

### 1) 필터 간 종속성의 가시성이 낮음 (지역 ↔ 조합)

- 관찰
  - 조합 옵션이 지역 선택에 따라 변경되지만, 사용자에게 "왜 옵션이 줄어들었는지" 설명이 없다.
  - 사용자가 지역을 변경했을 때 이전 조합 선택이 자동 제거될 수 있으나, 제거 사실이 명시적으로 드러나지 않는다.
- 사용성 리스크
  - 사용자는 "내가 선택한 조합이 왜 사라졌는지" 이해하지 못해 혼란을 느낄 수 있다.
  - 특히 분석 숙련도가 낮은 사용자에게 신뢰 저하로 연결된다.
- 개선 제안
  - 조합 필드 라벨 하단에 보조 문구 추가: "지역 선택에 따라 조합 목록이 자동 조정됩니다."
  - 지역 변경으로 조합이 제거된 경우, 2~3초 토스트/인라인 알림 제공: "선택 불가 조합 N개가 해제되었습니다."
  - 조합 필드 placeholder를 정적 "전체" 대신 동적 문구로 변경: "선택 가능한 조합 12개 중 전체"

### 2) 기간 필터의 오류 예방이 약함 (역전 범위, 빈 결과)

- 관찰
  - 시작일/종료일이 독립적으로 변경되어 역전 범위가 가능하다.
  - 역전 범위나 데이터가 없는 기간에도 즉시 원인 설명이 없다.
- 사용성 리스크
  - 사용자는 시스템 고장으로 오해할 수 있다.
  - "필터가 안 먹는다"는 인식으로 이어져 반복 사용 의지가 감소한다.
- 개선 제안
  - 입력 단계 예방: 종료일 DatePicker에 minDate=startDate, 시작일에 maxDate=endDate 적용
  - 자동 보정 정책(택1)
    - 정책 1: start > end가 되면 end를 start로 자동 이동
    - 정책 2: 즉시 에러 표시 후 적용 버튼 비활성화
  - 결과 단계 피드백: 0건일 때 빈 상태 문구에 원인 후보 제시
    - 예: "선택한 기간(2024-10-20 ~ 2024-10-01)에는 데이터가 없습니다. 기간 또는 지역/조합을 조정해 주세요."

### 3) 필터 적용 상태의 한눈 가시성 부족

- 관찰
  - 현재는 각 컨트롤 내부 상태를 직접 확인해야 필터 조합을 파악할 수 있다.
  - 모바일에서는 필터 영역이 접힌 상태에서 현재 적용 조건이 잘 드러나지 않는다.
- 사용성 리스크
  - 사용자는 "지금 어떤 필터가 적용 중인지" 기억에 의존하게 된다.
  - 인지 부하가 커져 실수와 재작업이 증가한다.
- 개선 제안
  - 필터 카드 상단에 "적용 중 필터 요약 바" 추가
    - 예: 지역 2개, 조합 3개, 기간 10/01~10/14, 등급 4개
  - 요약 바 내 각 칩에 제거(X) 액션 제공
  - 모바일 접힘 상태에서도 요약 바는 항상 노출

### 4) "초기화" 액션의 영향 범위가 직관적이지 않음

- 관찰
  - 초기화 버튼이 모든 필터(비교 모드 포함)를 리셋한다.
  - 버튼 라벨만으로는 영향 범위(비교 기간 포함)가 충분히 예상되지 않는다.
- 사용성 리스크
  - 실수 클릭 시 작업 손실(재입력 비용) 발생
  - 고빈도 분석 사용자일수록 불만이 커진다.
- 개선 제안
  - 버튼 텍스트를 "전체 초기화"로 명확화
  - 클릭 시 짧은 확인 단계 제공(옵션)
    - "지역/조합/기간/비교설정을 모두 초기화할까요?"
  - 대안으로 "필터만 초기화"와 "비교 포함 전체 초기화" 분리

### 5) 비교 모드의 정신 모델이 약함

- 관찰
  - 비교 모드 활성 시 프리셋이 자동 적용되지만, 어떤 규칙으로 계산됐는지 설명이 부족하다.
  - 사용자는 현재 메인 기간과 비교 기간의 관계를 한눈에 보기 어렵다.
- 사용성 리스크
  - 비교 수치 해석 오류 가능성 증가
  - 의도하지 않은 비교 구간으로 의사결정할 위험
- 개선 제안
  - 비교 모드 영역에 "현재 기간 vs 비교 기간" 한 줄 요약 고정 표시
  - 프리셋 선택 시 설명 툴팁 제공
    - 전년 동기: "시작/종료일의 연도만 -1"
    - 직전 1주: "현재 시작일 기준 직전 7일"
    - 시즌 시작부터: "해당 연도 8/1부터 현재 종료일까지"
  - 비교 기간이 비시즌/무데이터 구간일 때 경고 배지 제공

### 6) 모바일 사용성: 터치 정확도와 정보 밀도 균형 필요

- 관찰
  - 멀티셀렉트 칩, 날짜 입력, 비교 프리셋 칩이 좁은 폭에서 밀집된다.
  - 작은 칩(24px 높이)은 일부 사용자에게 터치 난이도가 있다.
- 사용성 리스크
  - 오탭으로 인한 재조작 증가
  - 필터 조작 시간이 데스크톱 대비 과도하게 증가
- 개선 제안
  - 모바일 브레이크포인트에서 프리셋 칩 높이/패딩 확대
  - 지역/조합 멀티셀렉트에 "선택 완료"형 바텀시트 UX 검토
  - 필터 열기/닫기 버튼 외에 "적용" CTA를 명시해 상태 인식 강화

### 7) 접근성(A11y) 보완 필요

- 관찰
  - 멀티셀렉트 + 칩 기반 UI는 스크린리더 맥락 전달이 어려울 수 있다.
  - 필터 변경 후 결과 업데이트가 발생해도 라이브 리전 알림이 없다.
- 사용성 리스크
  - 보조기술 사용자에게 현재 상태와 변경 결과가 충분히 전달되지 않음
- 개선 제안
  - 필터 요약 바를 aria-live polite 영역으로 제공
  - 필터 변경 후 "N개 레코드" 변화 텍스트를 스크린리더 친화적으로 갱신
  - 키보드 탐색 순서(지역→조합→등급→기간→비교) 일관성 유지

### 8) 성능 체감 UX: 로딩 인지 개선 여지

- 관찰
  - 기간 변경 시 데이터 로딩이 발생하나, 필터 컨트롤과 결과 패널 사이의 상태 연결이 약하다.
- 사용성 리스크
  - 느린 네트워크에서 "클릭이 반영됐는지" 불명확
- 개선 제안
  - 필터 변경 즉시 상단에 로딩 인디케이터(선형 progress) 노출
  - 결과 패널 스켈레톤/디밍으로 "갱신 중" 시각화
  - 로딩 실패 시 "재시도" 액션 포함 오류 메시지 제공

## 문제의 우선순위 (사용성 기준)

### P0 (즉시 대응)

- 비교 모드 기간 요약/프리셋 설명 보강

### P2 (중기 개선)

- 조합 옵션 데이터 기반 동적 생성

- 접근성 라이브 리전/키보드 탐색 최적화

  - 조합 오타/누락 수정
  - 기간 역전 방지
  - 빈 결과 안내문 강화
- 기대 효과
  - "필터가 이상하다" 피드백 급감

### 2단계: 상태 가시성 강화

- 목표
  - 현재 필터 상태를 기억이 아닌 화면에서 즉시 파악
- 작업
  - 필터 요약 바 도입
  - 비교 기간 요약/설명 라벨 추가
- 기대 효과
  - 재탐색 시간 단축, 실수 감소

### 3단계: 고급 UX 안정화

- 목표
  - 데이터 변화에 강한 필터 구조
- 작업
  - 데이터 기반 옵션화, A11y, 모바일 최적화
- 기대 효과
  - 유지보수 비용 절감, 다양한 사용자 환경 대응

## 측정 지표 (정량)

> 별도 이벤트 수집 인프라(Analytics 등) 도입 시 참고. 현재 Vercel 기본 배포 환경에서는 실측 불가.

- 필터 과업 성공률: 목표 조건 도달 비율 (+10%p 목표)
- 빈 결과 발생률: 오류성 0건(역전/불일치 기인) 50% 이상 감소
- 초기화 후 즉시 재설정률: 초기화 직후 30초 내 재입력 비율 (-15% 목표)

## 결론

현재 필터 기능의 핵심 문제는 단순한 데이터 오타 이슈를 넘어, "상태 가시성 부족"과 "오류 예방 부족"이 결합되어 사용자가 결과를 신뢰하기 어렵다는 점이다. 기능 정확성 패치(P0)만으로도 체감 품질은 빠르게 개선되며, 이후 요약 바/비교 설명/A11y 보강을 통해 분석 도구로서의 신뢰성과 반복 사용성이 크게 향상될 것이다.
